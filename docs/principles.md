# Kinē Product Principles

The canonical list of principles that guide every product decision. These are non-negotiable — any feature, change, or design must be assessed against them.

---

## Core Identity

**Kinē is a flexible system that removes resistance so women can make sustained fitness progress.**

It's not a coach. It's not a social platform. It's not a calorie tracker. It's a programming tool that adapts to your body and your life.

---

## The Principles

### 1. Built on Women's Physiology
Recovery, volume, and balance are calibrated for women's bodies — not men's training scaled down. Posterior chain (glutes, hamstrings, upper back) is the primary priority. Cycle-phase awareness is built in, not bolted on.

### 2. Guide, Don't Gate
Observe, don't grade. No scores, no verdicts, no pass/fail. Kinē provides information and suggestions — the user decides. We don't lock features behind performance metrics or shame users for missing sessions.

### 3. Closed Loop
No integrations, no social, no external data feeds. Self-report is the signal. This isn't a resource constraint — it's a design decision. External data creates noise and anxiety without improving programming decisions.

### 4. One Purpose Per Page
Every screen should have one singular purpose. If you're asking the user to do two things, it's two screens. Onboarding especially — one question, one decision, per step.

### 5. Progressive Autonomy
The goal is smarter use, not more use. As users learn to train, they should need Kinē less for the basics and more for the intelligence layer. We don't create dependency — we build competence.

### 6. Body Intelligence Is Subtle
Women's health awareness — cycle phases, conditions like PCOS and endometriosis, life stages like perimenopause — is a core capability, not a headline gimmick. The programme adapts silently through existing systems (exercise swaps, warmup mods, breathing cues, AI context). Marketing leads with "understands your body", not "tracks your cycle." Women are more than any single dimension of their health.

### 7. Messaging Reflects Reality
Never guilt. Never shame. Never use performative motivation ("crush it!", "no excuses!"). State what happened, suggest what's next. Training is hard — respect that.

### 8. AI Is Infrastructure, Not Identity
Users shouldn't think about AI when using Kinē. They should think about their program. The AI is the engine, not the steering wheel. Don't surface "AI" in the UI unless it adds user value.

### 9. Offline First
The app must work where people train — including gyms with no signal. Core session logging works without internet. AI generation requires connectivity but programs are cached.

### 10. Privacy by Default
Collect the minimum data needed. No tracking cookies. No analytics pixels. No social graphs. Cycle data is health data — treat it with appropriate gravity.

### 11. Simplicity Over Completeness
188 curated exercises > 1,400 random ones. Every feature should be deliberately included, not exhaustively included. If we can't do it well, we don't do it.

### 12. About You Is Minimal
Name, weight, height. That's it for personal info. We don't need your age, body fat percentage, or social media handles to program your training.

### 13. Per-Day Control
Users need per-day session length control, not just a global duration setting. Tuesday's 45-minute lunch session is different from Saturday's 90-minute gym session.

### 14. No Scores, No Streaks
Don't gamify training. Streaks create anxiety when broken. Scores reduce complex effort to a number. Show progress through actual metrics (weights lifted, sessions completed) — not points.

### 15. Session, Not Workout
Language matters. It's a "session," not a "workout." It's "training," not "exercise." This positions the user as someone doing structured, intentional work — not random movement.

### 16. Exercise Education, Not Exercise Commands
Every prescribed exercise should be learnable within the app. Cues, common mistakes, video reference. Don't just tell users what to do — help them understand why and how.

### 17. Rest Is Programming
Recovery days and rest periods are as intentional as heavy sets. The app should prescribe rest as confidently as it prescribes work. Deloads are autoregulated — triggered by the user's soreness, energy, and body signals — not scheduled on a fixed calendar. Fixed-schedule deloads can impair progress when the body doesn't need them.

### 18. Equipment Reality
Not everyone has a full gym. The app must generate excellent programs with limited equipment. A bodyweight-only user should get a program that's thoughtful, not a consolation prize.

### 19. Injuries Are Constraints, Not Disqualifiers
Having a bad knee doesn't mean you can't train legs. It means the app needs to route around the limitation intelligently. Injuries narrow the exercise pool — they don't stop programming.

### 20. The User Is the Authority
Kinē suggests. The user decides. They can swap exercises, skip sessions, ignore recommendations. The app should make the best suggestion possible, then respect the user's choice without judgment.

---

## Using These Principles

### Before building a feature:
1. Which principles does this feature serve?
2. Does it violate any principles?
3. If it conflicts with a principle, is there a version that doesn't?

### Before shipping copy:
1. Does it guide without gating? (Principle 2)
2. Is it free of performative motivation? (Principle 7)
3. Does it respect the user's intelligence? (Principle 20)

### Before adding data collection:
1. Is this the minimum data needed? (Principle 10, 12)
2. Is cycle data handled as health data? (Principle 10)
3. Does the user understand why we're asking? (Principle 4)
